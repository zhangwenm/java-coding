# systematic-debugging

系统化调试技能，遵循四阶段调试流程。

## 核心原则

```
没有根因调查就不修复
```

## 四阶段

### Phase 1: 根因调查

**修复前必做:**

1. **仔细阅读错误消息**
   - 不要跳过错误或警告
   - 通常包含准确解决方案
   - 完整阅读堆栈跟踪
   - 记录行号、文件路径、错误码

2. **一致地复现**
   - 能可靠触发吗？
   - 确切的步骤是什么？
   - 每次都发生吗？
   - 如不可复现 → 收集更多数据，不要猜测

3. **检查最近变更**
   - 什么变更可能导致这个？
   - Git diff、最近提交
   - 新依赖、配置变更
   - 环境差异

4. **在多组件系统中收集证据**
   
   **添加诊断工具:**
   ```
   对每个组件边界:
     - 记录进入组件的数据
     - 记录离开组件的数据
     - 验证环境/配置传播
     - 检查每层状态
   
   运行一次收集证据显示哪里坏
   然后分析证据识别失败组件
   ```

5. **追踪数据流**
   
   从坏值起源向上追踪:
   - 坏值从哪里起源？
   - 什么调用时传了坏值？
   - 一直追踪到找到源
   - 在源处修复，不要在症状处

### Phase 2: 模式分析

1. **找工作的例子**
   - 在同代码库找类似工作的代码
   - 什么工作的和坏的很像？

2. **对比参考**
   - 如在实现模式，完整阅读参考实现
   - 不要略读 - 每行都读
   - 充分理解模式后再应用

3. **识别差异**
   - 工作的和坏的区别是什么？
   - 列出每个差异，无论多小
   - 不要假设"那不可能有关系"

4. **理解依赖**
   - 这需要什么其他组件？
   - 什么设置、配置、环境？
   - 它做什么假设？

### Phase 3: 假设和测试

1. **形成单一假设**
   - 清晰陈述: "我认为 X 是根因因为 Y"
   - 写下来
   - 要具体，不要模糊

2. **最小化测试**
   - 做最小可能改变测试假设
   - 一次一个变量
   - 不要同时修复多个东西

3. **验证前继续**
   - 成功？→ Phase 4
   - 失败？→ 形成新假设
   - 不要在之上加更多修复

### Phase 4: 实现

1. **创建失败的测试用例**
   - 最简单的复现
   - 可能的话自动化测试
   - 修复前必须有

2. **实现单一修复**
   - 解决识别的根因
   - 一次一个改变
   - 不要"既然在这里"改进
   - 不要捆绑重构

3. **验证修复**
   - 测试现在通过？
   - 其他测试没坏？
   - 问题真的解决了？

4. **如修复不工作**
   - 停止
   - 数数：你尝试了多少修复？
   - 如 < 3: 返回 Phase 1，用新信息重新分析
   - **如 ≥ 3: 停止并质疑架构**

## 红旗 - 停止并遵循流程

- "先快速修复，以后调查"
- "试试改 X 看是否有效"
- "加多个改变，跑测试"
- "跳过测试，手动验证"
- "可能是 X，让我修那个"
- "我不完全理解但这可能有效"
- **"再试一次修复"（已尝试 2+ 次后）**
- **每个修复揭示不同地方的新问题**

## 快速参考

| 阶段 | 关键活动 | 成功标准 |
|------|----------|----------|
| **1. 根因** | 读错误、复现、检查变更、收集证据 | 理解什么和为什么 |
| **2. 模式** | 找工作例子、对比 | 识别差异 |
| **3. 假设** | 形成理论、最小测试 | 确认或新假设 |
| **4. 实现** | 创建测试、修复、验证 | Bug 解决、测试通过 |

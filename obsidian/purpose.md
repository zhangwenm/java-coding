# 知识库目标（purpose.md）

> LLM 在操作此知识库时必须先读本文件，用于判断"什么值得记、记在哪里、怎么写"。

---

## 这个知识库是谁的

软件工程师 + 架构师，技术栈以 Java 为主，兼涉 Python、前端等，负责多个业务域（manager / yuncang / lot / datacenter / base），
使用 Claude Code 作为主要 AI 编程助手，正在建立 AI 驱动的开发工作流。

---

## 知识库存在的目的

**核心问题**：跨会话的上下文丢失。

每次新开 Claude Code 会话，之前的决策背景、踩坑经验、架构思路全部消失。
这个知识库就是为了解决这个问题：**把隐性知识显性化，让 LLM 下次能接着上次的思路继续。**

具体目标：
1. **架构决策留痕**：为什么这么设计，当时有哪些选项，选了什么，原因是什么
2. **踩坑不重踩**：生产问题、技术调研、接口对接的经验教训
3. **工具链沉淀**：Claude Code、cmux、autocli 等 AI 工具的使用方法和最佳实践
4. **任务上下文管理**：长线开发任务的断点恢复（配合 cmux worktree）

---

## 什么值得记

✅ **记**：
- 架构决策和被否决的方案（特别是"为什么不这么做"）
- 生产踩坑（根因 + 修复方案 + 预防措施）
- 多次用到的工具配置和命令
- 技术调研的结论（不是过程，是结论和依据）
- 任务执行中途的断点快照

❌ **不记**：
- 可以从代码/注释直接读到的信息
- 官方文档已有的内容（记链接即可）
- 一次性的临时操作步骤
- 未经验证的猜测

## 各目录的写作意图

| 目录 | 写什么 | 不写什么 |
|------|-------|---------|
| `projects/arch-*` | 系统设计决策、模块边界、数据流 | 代码实现细节 |
| `projects/pitfalls-*` | 根因分析、修复方案、预防措施 | 只是症状描述 |
| `guides/` | 可复用的操作步骤，能直接执行 | 概念解释（那是文档的事）|
| `tasks/` | 任务断点快照，下次启动入口 | 已完成任务的详细过程 |
| `inbox/` | 未整理的碎片，待归档 | 长期存放 |

## 笔记生命周期

### 过期标记

笔记不主动删除，用 `status` 字段标记：

| status | 含义 |
|--------|------|
| `draft` | 写了但未验证 |
| `in-progress` | 对应任务进行中 |
| `done` | 已验证、稳定 |
| `stale` | 内容已过期，仅保留作历史参考 |

**何时标 stale**：代码重构导致架构描述失效、接口废弃、工具链切换。

### 置信度

frontmatter 可选加 `confidence` 字段，区分：
- `confidence: 已验证上生产` — 可直接信任
- `confidence: 设计阶段未落地` — 仅供参考，实际可能不同
- 不加字段 = 默认已验证

### 删除 vs 归档规则

- **不删**：有历史价值的决策记录（被否决的方案往往比采纳的方案更值得留）
- **标 stale**：内容过期但可能有参考价值
- **直接删**：只有 inbox/ 里的一次性碎片、已完成算法练习任务

## LLM 写笔记时的行为准则

1. **先问"值得记吗"**：对照上方"什么值得记"判断，不满足则不写
2. **写给下一个会话的 LLM 看**：假设读者不了解背景，但是个聪明的工程师
3. **结论前置**：第一句话就是核心结论，背景和过程放后面
4. **打交叉链接**：遇到相关笔记主动加 `[[链接]]`，不要孤立存在
5. **更新优于新建**：已有笔记能补充就补充，不要制造重复
6. **frontmatter 必填**：tags、date、project、status 四个字段不能缺

## 项目背景（帮助 LLM 理解业务上下文）

主要业务域（均为 Spring Boot + Maven，内网 Git 仓库）：

| 业务域 | 核心功能 | 常用子模块 |
|--------|---------|---------|
| manager | 管理后台、众包平台 | rw-backend, crowdsourcing-backend, workorder-backend |
| lot | IoT 聚合层，机器人调度 | robot-call-api, open-yuncang-api |
| datacenter | 监控告警、任务执行 | job-executor, monitor, data-collect |
| yuncang | 货柜、AI 管家 | cargotransport |
| base | Hessian RPC 实现层 | config, scene, task, user |
| common | Hessian RPC 接口定义 | webservice, client-* |

**Hessian RPC 注意**：common/webservice 定义接口，base 实现，manager/lot/yuncang 调用。
改接口必须三端同步，参考 `hessian-orchestrate` skill。

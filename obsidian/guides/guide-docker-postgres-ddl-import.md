# Docker Postgres DDL 导入避坑

**结论**：从其他环境导出的 PostgreSQL DDL 不能原样粘到当前环境执行。优先用 `docker exec optimus-postgres psql -U optimus -d res_atomization -c "..."` 一次性执行，并去掉旧 schema、owner、序列引用，把表建到当前默认 `public` schema。

## 已验证环境

```bash
docker exec optimus-postgres env | grep POSTGRES
```

本次现场确认：

```text
POSTGRES_DB=res_atomization
POSTGRES_USER=optimus
```

因此统一使用：

```bash
docker exec optimus-postgres psql -U optimus -d res_atomization -c "SQL"
```

不要用：

```bash
docker exec optimus-postgres psql -U postgres ...
```

该容器内没有 `postgres` 角色，会报：

```text
FATAL: role "postgres" does not exist
```

## 核心改造规则

### 规则 1：去掉导出环境 schema

导出 SQL 里常见：

```sql
CREATE TABLE "l2_dev"."intent" ...
CREATE TABLE "l3_test2"."t_task" ...
CREATE TABLE "provider_dev"."service_provider" ...
```

当前程序日志查询的是裸表名，例如：

```sql
FROM l2_exec_task_run
INSERT INTO l2_exec_task_step ...
```

所以导入时建到默认 schema：

```sql
CREATE TABLE IF NOT EXISTS intent ...
CREATE TABLE IF NOT EXISTS t_task ...
CREATE TABLE IF NOT EXISTS service_provider ...
```

否则会遇到：

```text
ERROR: schema "l2_dev" does not exist
ERROR: schema "res_atomization" does not exist
```

注意：`res_atomization=#` 是数据库名提示符，不代表存在 `res_atomization` schema。

### 规则 2：去掉 OWNER TO

导出 SQL 常见：

```sql
ALTER TABLE "provider_dev"."service_provider" OWNER TO "l4_dev";
```

当前库不一定有这个 role，执行会报：

```text
ERROR: role "l4_dev" does not exist
```

一般跳过 owner 修改即可。表能否被程序使用取决于当前连接用户权限，不取决于导出环境 owner。

### 规则 3：把 nextval 序列改成 identity 或 bigserial

导出 SQL 常见：

```sql
"id" int8 NOT NULL DEFAULT nextval('"l2_dev".intent_id_seq'::regclass)
```

当前 schema 下没有这个 sequence。改成：

```sql
id bigserial NOT NULL
```

或：

```sql
id int8 NOT NULL GENERATED BY DEFAULT AS IDENTITY
```

### 规则 4：索引名要避免重复

PostgreSQL 同一个 schema 下索引名不能重复。导出环境里的通用名可能冲突：

```sql
CONSTRAINT "uk_code" UNIQUE ("code")
```

如果报：

```text
ERROR: relation "uk_code" already exists
```

改成表名前缀：

```sql
CONSTRAINT uk_service_provider_code UNIQUE (code)
```

或：

```sql
CREATE UNIQUE INDEX IF NOT EXISTS uk_service_provider_code ON service_provider (code);
```

### 规则 5：长 SQL 不要在 psql 交互里粘贴

本次多次出现交互粘贴丢字符：

```text
CONSTRAINT -> NSTRAINT
now() -> ow()
```

导致：

```text
ERROR: syntax error at or near "uk_intent_intent_id"
ERROR: syntax error at or near "6"
```

优先在 shell 执行：

```bash
docker exec optimus-postgres psql -U optimus -d res_atomization -c "CREATE TABLE ..."
```

如果必须用 heredoc，结尾 `SQL` 必须单独一行且前面不能有空格，否则终端会像卡死一样等待输入。

## 标准命令模板

```bash
docker exec optimus-postgres psql -U optimus -d res_atomization -c "CREATE TABLE IF NOT EXISTS table_name (...); CREATE INDEX IF NOT EXISTS idx_name ON table_name (...);"
```

验证：

```bash
docker exec optimus-postgres psql -U optimus -d res_atomization -c "\d table_name"
```

查表是否存在：

```bash
docker exec optimus-postgres psql -U optimus -d res_atomization -c "\dt"
```

查当前 schema：

```bash
docker exec optimus-postgres psql -U optimus -d res_atomization -c "SHOW search_path;"
```

## 常用改造示例

### l2_exec_task_run

```bash
docker exec optimus-postgres psql -U optimus -d res_atomization -c "CREATE TABLE IF NOT EXISTS l2_exec_task_run (task_id text NOT NULL, intent_name text, intent_params jsonb, resources jsonb, status text NOT NULL, chassis_id text, started_at timestamptz(6), ended_at timestamptz(6), error_code text, error_message text, created_at timestamptz(6) NOT NULL DEFAULT now(), updated_at timestamptz(6) NOT NULL DEFAULT now(), CONSTRAINT l2_exec_task_run_pkey PRIMARY KEY (task_id)); CREATE INDEX IF NOT EXISTS idx_l2_exec_task_run_chassis ON l2_exec_task_run (chassis_id); CREATE INDEX IF NOT EXISTS idx_l2_exec_task_run_status_created ON l2_exec_task_run (status, created_at DESC NULLS FIRST);"
```

### l2_exec_task_step

```bash
docker exec optimus-postgres psql -U optimus -d res_atomization -c "CREATE TABLE IF NOT EXISTS l2_exec_task_step (id bigserial NOT NULL, task_id text NOT NULL, seq int4 NOT NULL, operator_name text NOT NULL, goal_id text, step_path text, params jsonb, outputs jsonb, success bool, error_code text, started_at timestamptz(6) NOT NULL, ended_at timestamptz(6), duration_ms int4, CONSTRAINT l2_exec_task_step_pkey PRIMARY KEY (id)); CREATE INDEX IF NOT EXISTS idx_l2_exec_task_step_task_seq ON l2_exec_task_step (task_id, seq);"
```

### l2_exec_task_step_call

```bash
docker exec optimus-postgres psql -U optimus -d res_atomization -c "CREATE TABLE IF NOT EXISTS l2_exec_task_step_call (id bigserial NOT NULL, task_id text NOT NULL, step_id int8, seq int4 NOT NULL, call_type text NOT NULL, target text, request jsonb, response jsonb, sent bool, completed bool, timeout bool, error text, started_at timestamptz(6) NOT NULL, ended_at timestamptz(6), duration_ms int4, CONSTRAINT l2_exec_task_step_call_pkey PRIMARY KEY (id)); CREATE INDEX IF NOT EXISTS idx_l2_exec_task_step_call_step ON l2_exec_task_step_call (step_id); CREATE INDEX IF NOT EXISTS idx_l2_exec_task_step_call_task ON l2_exec_task_step_call (task_id, started_at);"
```

### service_provider

```bash
docker exec optimus-postgres psql -U optimus -d res_atomization -c "CREATE TABLE IF NOT EXISTS service_provider (id int8 NOT NULL GENERATED BY DEFAULT AS IDENTITY, code varchar(64) NOT NULL, name varchar(64) NOT NULL, auth_type varchar(64) NOT NULL, status int2 NOT NULL DEFAULT 1, created_at timestamp(6) NOT NULL DEFAULT CURRENT_TIMESTAMP, CONSTRAINT service_provider_pkey PRIMARY KEY (id), CONSTRAINT uk_service_provider_code UNIQUE (code)); COMMENT ON COLUMN service_provider.code IS '提供商编码，如：yunji_robot'; COMMENT ON COLUMN service_provider.name IS '提供商名称'; COMMENT ON COLUMN service_provider.auth_type IS '鉴权类型'; COMMENT ON TABLE service_provider IS '第三方服务提供商表';"
```

### provider_credential

如果表已存在但字段不完整，先用 `\d provider_credential` 确认结构，再补字段：

```bash
docker exec optimus-postgres psql -U optimus -d res_atomization -c "ALTER TABLE provider_credential ADD COLUMN IF NOT EXISTS name varchar(128) DEFAULT NULL, ADD COLUMN IF NOT EXISTS credential_data bytea, ADD COLUMN IF NOT EXISTS status int2 NOT NULL DEFAULT 1, ADD COLUMN IF NOT EXISTS expire_at timestamp(6), ADD COLUMN IF NOT EXISTS created_at timestamp(6) NOT NULL DEFAULT CURRENT_TIMESTAMP, ADD COLUMN IF NOT EXISTS updated_at timestamp(6) NOT NULL DEFAULT CURRENT_TIMESTAMP; ALTER TABLE provider_credential ALTER COLUMN credential_data SET NOT NULL; COMMENT ON COLUMN provider_credential.provider_id IS '服务提供商ID'; COMMENT ON COLUMN provider_credential.store_id IS '门店ID（仅当作用域为STORE时非空）'; COMMENT ON COLUMN provider_credential.name IS '凭证备注名称'; COMMENT ON COLUMN provider_credential.credential_data IS '加密后的凭证数据'; COMMENT ON COLUMN provider_credential.status IS '0-失效,1-有效'; COMMENT ON TABLE provider_credential IS '第三方服务凭证实例表';"
```

本次验证结构里有一个重要差异：

```text
store_id character varying(128)
```

原导出 DDL 是 `int8`。如果业务使用 `space_019e...` 这种字符串 storeId，不要强行改成 bigint。

## 从日志判断缺表

L2 Execution 缺运行表时会报：

```text
relation "l2_exec_task_run" does not exist
relation "l2_exec_task_step" does not exist
```

这类错误通常不会阻断 HTN 执行，但会导致任务运行记录、步骤记录、审计和恢复能力缺失。

Orchestrator 如果报：

```text
database "optimus" does not exist
```

这是连接库名配置问题，不是缺表问题。需要检查 Orchestrator 数据库连接配置是否仍指向旧库名 `optimus`。

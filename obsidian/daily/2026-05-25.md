## 编码记录

### 方案设计关键决策

1. **Hessian 代理 vs 直接实现** → 选 Hessian 代理，SceneAccountService 已配置，零数据源改动
2. **create 使用 createTmpUser** → robot-api 无 web session，用 appname 替代 ticket
3. **UserManagerService 是本地 Bean 非 Hessian** → 照抄 userAuthService 模式，配在 application-bean.xml
4. **三层安全**：ValidationFilter SN 签名(已有) → 设备-地点归属校验(新增) → ES 审计日志(新增)

### 5 个接口

| 操作 | 路径 | 鉴权 |
|------|------|------|
| 查询 | GET /api/v2/place/account/query | SN 签名 + 归属校验 |
| 新建 | POST /api/v2/place/account/create | SN 签名 + 归属校验 + createTmpUser |
| 取物码 | POST /api/v2/place/account/code/update | SN 签名 + 归属校验 |
| 编辑 | POST /api/v2/place/account/update | SN 签名 + 归属校验 |
| 移除 | POST /api/v2/place/account/remove | SN 签名 + 归属校验 |

## 学到的东西
- UserManagerService 在 rw-backend 中是本地 Bean（application-bean.xml），非 Hessian，继承 HttpService 通过 HTTP 调 SSO REST API
- lot/robot-api 鉴权仅依赖 ValidationFilter 的 SN 签名（productId+ts+sign→MD5），无 RBAC
- createTmpUser(appname, ...) 可替代 create(ticket, ...) 做服务间调用

- 读代码要先读完整文件，`sed -n` 截片段容易拿到错误方法的内容，导致整条分析链路崩塌
- 代码分析中的"怀疑"要尽量追到可确定的证据：`t_serial_number.type` 存 `"WTH"` 可从 `ConfigRoServiceImpl` 的常量集合和 `PadSchemeApi` 的防御性覆盖逻辑推断，不需要查库
- robot-api 的 `sceneBaseService` 走 Hessian（`hessian-bean.xml`），`padConfigService` 也走 Hessian，不是本地实现也不是旧 HTTP 客户端

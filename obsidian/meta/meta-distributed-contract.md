# 分布式契约变更必须原子化

**结论**：任何跨进程契约（接口、消息格式、数据库 Schema）的变更，必须一次性同步所有相关方，不能只改一侧。

## 适用场景

- RPC 接口变更（Hessian / gRPC / Feign）
- Kafka 消息 Payload 格式变更
- 数据库表结构变更（多服务共享同一 DB）
- HTTP API 字段增删

## 核心规则

| 操作 | 正确做法 | 错误做法 |
|------|---------|---------|
| 新增字段 | 接口定义 + 实现方 + 所有调用方同步部署 | 只改接口定义，调用方等下次发版 |
| 删除字段 | 先废弃（保留旧字段），等所有调用方去除依赖后再删 | 直接删字段 |
| 修改类型 | 新字段并存旧字段，灰度迁移 | 原地改类型 |

## 为什么不能只改一侧

序列化框架（Hessian/Protobuf/JSON）在字段不匹配时行为不一致：
- Hessian：字段缺失时反序列化为 null，不报错，导致静默数据错误
- JSON（Jackson）：未知字段默认忽略，但缺失必填字段会 NPE
- 两侧版本错位窗口期内，线上可能同时跑新旧两个版本

## 检查清单

- [ ] 找到所有调用方（grep 接口名）
- [ ] 确认部署顺序（通常：接口定义 → 实现方 → 调用方）
- [ ] 验证序列化兼容性（新旧字段共存时的 null 处理）
